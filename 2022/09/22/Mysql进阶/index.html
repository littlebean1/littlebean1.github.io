Mysql进阶

发表于 2022-09-22 更新于 2023-09-27 分类于 技术

mysql 进阶

mysql5.7 和 mysql8.0在编码方面的区别

mysql5.7:

底层编码采用latin1，在输入汉语时会发生乱码，如果要存储汉语需要修改编码规则，修改编码后表中已经存在的数据需要删除再导入

mysql8.0

底层编码采用utf8m4，输入汉语时不会发生乱码

mysql的逻辑结构

  • 连接层：建立TCP连接，三次握手后，客户端与数据库连接池链接
  • 服务层：接受mysql指令，进行语法、词义分析，构建语法树，对sql进行优化
  • 引擎层：与底层的文件系统交互，cpu将文件系统中的数据加载到内存中

存储引擎：

数据库表在底层的存储类型，mysql8的默认存储引擎为InnoDB

Memory引擎

  • 所有数据都存在内存中，读写效率很高，但是MySQL服务重启后，数据会丢失
  • 它不支持事务
  • memory支持hash索引或者B树索引

InnoDB引擎

  • 事务存储引擎，支持事务

  • 提供行锁，操作时只锁某一行，适合高并发的操作

  • 以更新删除为主时，优先选择InnoDB

  • 缓存索引与真实数据，对内存要求高—(聚簇索引)

MyIsam引擎

  • 非事务处理存储引擎，不支持事务

  • 提供表锁，不支持行锁，操作表中一条记录也会锁住整个表，不适合高并发

  • 以增加和查询为主时，优先选择MyIsam

  • 只缓存索引，不缓存真实数据—(非聚簇索引)

  • xxx.sdi文件存储表结构信息，xxx.MYD存储数据，xxx。MYI存储索引

  • 不能使用外键

索引

索引分类

  • 按数据结构分类： hash索引 (不可范围查询，排序)，B树索引(层级高，范围查询不如B+树方便)，B+树索引
  • 按照物理存储分类：聚簇索引，非聚簇索引
  • 按照字段特性分类，普通索引，主键索引，唯一索引
  • 按照字段个数分类，单列索引，联合索引

hash索引的特点

  • 通过hash函数将key转为hash值映射到槽位上，每个槽位有一个链表存储hash冲突的数据
  • 只能用于简单的比较，不能进行范围查询
  • 数据的存储是无序的，使用order by时需要重排序

B+树

​ B+树种结点占用一个页16KB，非叶子结点不存储数据，只存储主键值和页面索引 (费聚簇索引格外存储索引列)，叶子结点存储真实数据。B+树的各个叶子结点之间形成一个双向链表，叶子结点内部是一个单向链表。此外B+树结点只存储索引项，但是B树结点需要存储数据浪费空间导致可一个页可索引的数目少，而且B+树通过链表查找速度快。

为什么B+树一般需要三层

​ InnoDB存储引擎中页的大小为16KB ，假设一个索引项长为16B，则三层B+树可以存储1000 * 1000 * 1000个索引项，实际情况中，每个节点可能填不满，因此B+树的高度个能达到3-4层。Mysql的InnoDB存储引擎将根节点常驻内存(根节点的地址是永恒不变的，每次扩充根节点地址始终是最上层)，因此要查找某一键值需要2-3次磁盘I/O操作。Mysql底层数据结构采用B+树，B+树的一个结点大小为16K，所以mysql一页大小为16k，B+树的叶子结点存储数据页，非叶子结点存储索引页。一页中有多个行数据，一行包括记录字段，roll_pointer，trx_id，row_id(没有主键时自动生成)。

B树与B+树的区别

  • B树种所有的结点都存储真实数据，B+树只有叶子结点存储真实数据，非叶子结点存储索引项
  • B树由于页中存储真实数据，占用空间大，页中存储的节点数量有限，一个页可索引的数目少，B+树页中只存储索引项，一个页中支持大量索引16KB/16B=1000
  • B树在查询数据时，比如a>10查询复杂，而B+树的所有叶子结点都是通过双向链表连接的，查询数据方便
  • B树一个页面中n个关键字对应n+1个子节点，B+树的孩子结点的数量等于节点内关键字的数量，因为每个关键字存储着一个页号，页号对应一个页面

B+树的缺点

  • 如果主键不是递增的，会导致每次插入数据产生大量的数据迁移，影响性能
  • 即使主键是有序的，随着新数据的插入叶子结点会慢慢分裂，分裂后的数据可能在磁盘上相隔很远，会产生大量的随机IO

影响B+树磁盘IO的因素，B+树每一层越宽越好吗

  • mysql每一页的大小也就是B+树的节点大小，当页越大时，一个页存储的索引数目越多，每一层能够存放的页也就越多，在数据量一定的情况下，页越大，B+树越低，磁盘IO越少
  • 不是越宽越好，如果B+树节点过大，会导致内存无法一次性加载一个节点，导致额外的磁盘IO操作，降低性能。

单列索引：

​ 又称单列索引，在普通的字段上添加索引

唯一索引

​ 添加唯一约束时，会自动创建唯一索引

全文索引

​ 在查询少量varchar类型数据时，使用like模糊查询很方便，当查询大量varchar类型的数据时，like模糊搜索效率很低。采用全文索引查询速度比like快很多。全文索引必须建立在字符串上，在查询时进行等值匹配，通过倒排索引来实现，对数据进行分词处理，将一个数据分成多个词，然后建立一张表，存储被分词分出的单词并做统计。

聚簇索引 (指针占用6B)

​ 表中的记录与数据存储处在一起，以主键id为索引，整个B+树存储在.ibd文件中

一个非主键字段

二级索引 (联合索引，非聚簇索引)

​ 叶子节点中存储的是索引的字段+主键，通过主键去聚簇索引中找到需要的值 (回表)

联合索引的注意事项

​ 将查询需求频繁或者字段选择性高的列放在索引的前面。

​ 为经常需要排序、分组的字段建立索引

​ 连接查询有索引的表当被驱动

哪些情况下不应该使用索引

  1. 数据唯一性差的字段不要使用索引，回表次数过多(访问磁盘)
  2. 频繁更新的字段不要使用索引
  3. 字段不在where语句出现时不要添加索引
  4. 数据量少的表不要使用索引

MyIsam

​ myisam的索引方式都是非聚簇索引，这意味着myisam中建立的索引全部都是二级索引，都需要回表操作，但是与InnoDB不同的是，innoDB的回表是拿主键到聚簇索引找到对应的记录；myisam的回表拿到的是对应记录的地址，查询速度很快。

回表或者二次查询

​ 使用聚簇索引查询可以直接定位到记录，而非聚簇索引(不包括覆盖索引)通常需要扫描两遍索引树，即先通过非聚簇索引索引定位到主键值，在通过聚簇索引定位到行记录，这就是所谓的回表查询，它的性能比扫描一遍索引树低。

覆盖索引

​ 一个索引包含了满足查询结果的数据叫覆盖索引。在建立非聚复合索引时，包括这整个sql语句的所有属性列都在整个 (索引+主键) 中时，可以直接从B+树叶子结点中取出数据，不需要回表。有主键是因为B+树结点必然包含着主键列，回表是通过主键来完成的

​

索引失效

  • where语句中有计算，函数，类型转换会导致索引失效

  • 最左匹配原则，检索数据时从联合索引的最左侧开始匹配 ，where子句搜索条件顺序调换不影响索引使用，因为查询优化器会自动优化查询顺序

    • 在 InnoDB 中联合索引只有先确定了最左侧的值后，才能确定下一个值。如果有范围查询的话，那么联合索引中使用范围查询的字段后的索引在该条 SQL 中都不会起作用，in 和 = 都可以乱序，查询优化器会按照定义索引的顺序 a,b,c ，重排where b>0 and a=1 and c=1语句按照 a,b,c

  • 范围条件右侧的列索引都失效，INDEX(age,id,name) where age=30 and id>20 and name=’abc’，命中age和id索引

  • like以通配符%开头

  • or前面和后面存在非索引的列

  • 联合查询时，驱动表索引无效，被驱动表索引有效 select * from a(驱动表) left join b (被驱动表) on a.id=b.id

  • in通常是走索引的，当in后面的数据在数据表中超过30%的匹配时，会走全表扫描，即不走索引，因此in走不走索引和后面的数据有关系。

主键索引和唯一索引的区别

  • 主键不允许有空值，唯一键可以有空值
  • 主键是一个唯一性索引，唯一性索引不一定是主键
  • 主键可以被其他表引用为外键，唯一键不可以

事务的ACID原理

事务与隔离级别

​ 事务是一系列操作的集合，是一个执行单元。

​ 事务的四个特性：原子性，一致性，隔离性，持久性

​ 隔离级别：脏读(未提交读取)，不可重复读(delete，update，针对某条数据，多次读取信息不一致)，幻读(insert，针对大量信息，多次读取数据数目不一样)

​ 在Mysql中，默认的隔离级别是可重复读，由于mysql采用MVCC的机制，所以可重复度解决了幻读的问题

  • 隔离性通过加不同的锁来实现；

  • 原子性通过undo log实现，undo log用于事务的回滚和MVCC，存储于磁盘中

  • 持久性通过redo log实现，redo log用于数据库崩溃后数据恢复，存储于磁盘中

  • bin log记录修改表结构和表中数据的SQL语句，用于集群中备份数据

redo log

​ 重做日志：记录的是事务提交时数据页的物理修改，用来实现事务的持久性

​ 该日志由两部分组成，重做日志缓冲(redo log buffer，位于内存中)和重做日志文件(redo log file，位于磁盘中)。

​ 刷盘策略：

  • 当事务提交之前会把所有修改信息都存到redo log buffer中，当事务提交后把redo log buffer中修改后的数据发送到page cache，由操作系统将page cache数据再刷盘到redo log file文件中
  • redo log buffer中的数据每隔1s被刷到redo log file文件上
  • mysql关闭时，将redo log buffer中的数据刷到redo log file文件上

undo log

​ 回滚日志：用于记录数据修改前的信息，提供回滚和MVCC，用来实现事务的原子性和一致性

​ 该日志是逻辑日志 ( 一系列操作时真真切切发生了，只不过在逻辑上相互抵消了而当做没发生 )，每当delete一条数据时，undo log会记录一条对应insert记录，updata同理。当执行回滚时，从undo log中的逻辑记录中读取对应的内容进行回滚。undo log在事务执行时产生，事务提交时并不会立即删除undo log，因为可用于MVCC

​ undo log分为insert undo log和update undo log：

  • insert undo log是指在insert操作中产生的undo log，因为在可重复读的隔离级别下，insert操作的记录，只对事务本身可见，其他事务不可见，该undo log在事务提交后可以直接删除

  • update undo log是指在update和delete操作中产生的undo log，该undo log可能需要提供MVCC机制所需要的版本链，因此在事务提交时不能立即删除，等待其他的线程删除

三大日志执行流程

MVCC

innodb引擎中每个行记录都含有隐藏字段：

  • DB_TRX_ID：记录插入这条记录或最后一次修改该记录的事务ID
  • DB_ROLL_PTR：回滚指针，配合undo log，指向这条记录的上一个版本
  • DB_ROW_ID：隐藏主键，如果表中无主键，则会添加该字段

版本链：

ReadView的组成结构：

  • trx_id：事务的id

  • trx_ids：已执行但还未提交的事务的列表，[10,20]

  • up_limit_id：trx_ids表中最小的事务id，表示活跃事务的id下限

  • low_limit_id：事务id不能超过所有事务中id的最大值+1

在不可以访问的情况下，需要查询undo log的版本链，向前找到可见的记录并返回

可重复读隔离级别下的MVCC

​ 事务在执行第一条select语句时生成新的ReadView，此时ReadView的trx_ids列表内容为[10, 20]，从undo log的版本链中找出可见的记录；当该事务后面的select语句再次读取同一数据时，直接复用之前生成的ReadView，也就是说，如果在执行第一条select后有其他事务已经提交，所读取的数据仍然是第一次select读取的结果。

读已提交隔离级别下的MVCC

​ 事务在执行第一条select语句时生成新的ReadView，此时ReadView的trx_ids列表内容为[10, 20]，从undo log的版本链中找出可见的记录；当该事务后面的select语句再次读取同一数据时，重新生成新的ReadView，在该select语句执行之前，若有事务执行完成，则根据undo log版本链可以找到已提交的事务的更新的数据

锁的粒度

​ InnoDB支持多粒度锁，行级锁可以和表级锁可以共存，意向锁在保证并发性的前提下，实现了行锁和表锁的共存

表锁

  • 表级别的S锁、X锁
  • 意向锁
  • 自增锁
  • 元数据锁

​ 当对表中某一行添加排他锁时，会默认给表添加一个意向排它锁；对表中两条不同的数据添加行级锁时，意向排他锁是可以兼容的。意向锁的作用是当别的事务想要获取表级锁的时候不需要去扫描全表来查看是否有行级锁

行锁

  • 记录锁
  • 间隙锁
  • 临键锁
  • 插入意向锁

​ 记录锁：分为S型记录锁和X型记录锁，针对已存在的记录。当一个事务获取了一条记录的S型记录锁后，其他事务也可以继续获取该记录的S型记录锁，但不可取X型记录锁。当一个事务获取了一条记录的X型记录锁后，其他事务既不可以继续获取该记录的S型记录锁，也不获取X型记录锁。

​ 间隙锁：针对不存在的记录，该锁是为了防止插入幻影记录而提出的。当我们在访问某条不存在的记录时，会对该记录主键所位于的区间添加间隙锁，如果后来其他本事务或事务执行insert操作，且主键位于间隙锁的区间内，则会阻塞insert操作。

​ 记录锁和间隙锁可以同时用，并不冲突。如果对一条记录加了gap锁，不会限制其他事务对该记录加记录锁

间隙锁、临键锁、记录锁

临键锁 = 间隙锁+记录锁 <=> (8,16] = (8,16)+[16]
记录锁：行锁，防止其它事务 新增/更新/删除 该行
间隙锁：锁住一个区间，防止其它事务修改该区间的记录，可读

select * from user where userId = 16 for update;

  • 唯一索引等值查询
    • 查询记录存在：先添加临键锁(8,16]，查到记录存在，临键锁退化为记录锁，id = 16加记录锁
    • 查询记录不存在：先添加临键锁(8,16]，查到记录不存在，临键锁退化为间隙锁， (8,16)加间隙锁
  • 非唯一索引等值查询
    • 查询记录存在：查到满足条件的还不能停，需要找到下一个不满足的为止 (可能会有多个等值)，加临键锁和间隙锁
    • 查询记录不存在：临键锁退化为间隙锁

在RR级别下，解决幻读的方式有两种：

  • 快照读(读取旧数据)是通过MVCC和undo log来实现的
  • 当前读(读取最新数据)是通过加记录锁和间隙锁来实现的。

MySQL中select语句的执行过程

  • 连接器：TCP三次握手建立连接，校验用户身份

  • 查询缓存：在缓存中查找是否存在结果,如果存在直接返回,否则继续执行.MySQL8.0删除了查询缓存

  • 解析器：经过词法分析得到SQL语句中的关键字并构建SQL语法树.根据语法分析分析SQL语句是否符合MySQL语法

  • 优化器：在开始执行SQL之前，优化器根据表中的索引选择使用哪个索引，选择最优方案。

  • 执行器：

    • 预处理阶段：判断表和字段是否存在,将*扩展为表上所有列
    • 执行阶段：执行SQL语句，从存储引擎中读取数据返回给客户端.

truncate和delete的区别

truncate table 会删除表中所有的数据，自增主键重置为1，当表中有外键的时候需要先删除外键否则删除数据失败，truncate table操作无法回滚

delete from table 可以删除表中部分数据，可以删除所有数据，delete表数据可以回滚

limit分页查询100w条数据效率低

  • select userId,userName,address from user order by userName limit 1000,10
  • 该语句的执行效率低，添加索引idx_userName可以提高查询效率
  • select userId,userName,address from user order by userName limit 10000,10
  • 该语句的执行效率低，且不会命中索引，因为全表扫描的速度会比索引扫描更快。在读取第100000-100010条数据时，会对前100010条数据回表，然后丢弃掉前100000条数据。
  • select userId,userName,address from user u1 join (select userId from user order by userName limit 10000,10) u2 on u1.userId = u2.userId
  • 采用手动回表的方式，即在分页查询数据时先查出前100010条数据的userId，因为userId是主键，所以查询不会回表。查询到userId后再通过连表查询出需要回表的数据。需要注意分页的数据不要过多否则会影响效率

Mysql主从同步原理-GTID

1）当主数据库进行数据更新时，会在事务前产生GTID号，一同记录到binlog日志中。

2）从数据库端的I/O线程将变更的binlog数据，写入到本地的中继日志（relay log）中。

3）从数据库端的SQL线程从中继日志中获取GTID号，然后对比本地的Binlog查看是否有记录。如果有则说明该GTID的事务已经执行，此时从数据库会忽略。

4）如果没有记录，则从数据库就会从中继日志中获取数据并执行该GTID的事务，并记录到binlog中。根据GTID号可以知道事务最初是在哪个数据库上提交的。

MySQL添加新字段原理

​ 在向一个已有的数据表中添加新字段时，MySQL会将所有数据行都复制到一个新的数据表中，然后添加新字段，最后再将数据行复制回原来的数据表中。这个过程可能需要花费很长时间，而且会占用大量的系统资源。建议在非高峰期执行这个操作，以避免影响系统的正常运行。数据文件idb在磁盘上是一行一行顺序存储的，添加新字段时数据文件不易操作，所以需要借助一个临时数据表
