JUC并发编程

发表于 2022-12-09 更新于 2023-09-03 分类于 面试题

多线程锁

并发中同步的方法

  • wait(),notify() Object
  • await(),signal() Object
  • yield()：Thread类，使该线程放弃CPU，回到就绪态，但不释放锁不会进入阻塞状态，等待CPU的调度
  • join()：t.join()，当前线程会等待t线程运行结束后继续运行，底层调用了当前线程的wait()方法，wait()使用必须获得锁，因此该方法时被Synchronized修饰的，t线程执行完毕后执行notify()，当前线程拿到锁继续执行

线程池

线程池参数

在初始化线程池参数时应注意

  • 根据时CPU密集型还是IO密集型设置对应的线程数

  • 根据应用场景选择对应的抛弃策略，比较关键的业务，准确性高，推荐使用AbortPolicy拒绝策略，这样子在系统不能承载更大的并发量的时候，能够及时的通过异常发现

  • 选择合适的线程池阻塞队列

    • 如果是有高峰期和低谷期的业务，不宜采用ArrayBlockingQueue，该队列不能够扩容，可以考虑LinkedBlockingQueue
    • 如果是比较平稳的业务，可以考虑使用ArrayBlockingQueue，该队列不像队列一样需要使用额外的指针空间，内存利用率更高

线程池工作原理

  • 线程池创建完成后，等待任务请求 (创建线程池时线程数量为0，状态为RUNNABLE)

  • 调用execute()方法时会将创建线程执行任务 execute()方法的执行流程

    • 如果正在运行的线程数量少于核心线程数，则来一个新任务创建一个线程
    • 如果正在运行的线程数量等于核心线程数，但是阻塞队列没满，则放入队列
    • 如果正在运行的线程数量大于核心线程数且小于最大线程数，而且阻塞队列已满，则创建非核心线程运行任务
    • 如果阻塞队列已满且线程数量等于最大线程数，线程池采用拒绝策略来处理任务
      • AbortPolicy 新任务提交时直接抛出未检查的异常RejectedExecutionException，该异常可由调用者捕获，默认策略
      • DiscardPolicy 丢弃该任务，适用于对任务丢失不敏感的场景,如点赞和浏览量等
      • DiscardOldestPolicy 丢弃阻塞队列中等待最久的任务，适用于对新任务优先级较高的场景
      • CallerRunsPolicy 将这个任务交于提交任务的线程执行，适用于希望调用者自己处理被拒绝的任务的场景

  • 线程的任务完成则将使用的线程放入线程池，执行其他的任务

  • 当某个线程空闲且等待时间超过存活时间时

    • 如果线程池中线程数大于核心线程数，则停掉该线程
    • 如果线程池中线程数等于核心线程数则保持存活状态

线程池状态

  • RUNNABLE：线程池一旦被创建，就处于RUNNING状态，且线程池中的任务数为0，线程池可以正常接收和处理任务
  • SHUTDOWN：调用线程池的shutdown()方法时，线程池由RUNNING->SHUTDOWN。该状态不接收新任务，但可处理已添加的任务 (阻塞队列中)
  • STOP：调用线程池的shutdownNow()方法时，线程池由RUNNING->SHUTDOWN。不接收新任务，不处理已添加的任务，中断正在处理的任务
  • TIDYING：当所有任务完成或终止后，线程池由(SHUTDOWN/STOP)->TIDYING状态
  • TERMINATED：线程池处于TIDYING状态后，调用钩子函数terminated()，线程池由TIDYING->TERMINATED，所有线程都已销毁，线程池停止工作

阻塞队列

  • LinkedBlockingQueue：无界队列，任务最大为Integer.MAX_VALUE，可能会OOM，用于FixedThreadPool和 SingleThreadExector
  • SynchronousQueue：同步队列，只要没有空闲线程就创建线程

线程池类别

​ 该类为我们提供了线程工厂，以及一些设定好参数的线程池可以直接使用

  • newFixedThreadPool //固定线程数
  • newSingleThreadExecutor //单线程
  • newCachedThreadPool //线程池中核心线程数为0，非核心线程上限为Integer_MAX

创建线程的三种方式

  • 实现Runnable接口->thread.start()->createThread.run()
  • 继承Thread类->createThread2.start()->thread.start()->createThread2.run()
  • 实现Callable接口->thread.start()->futureTask.run()->call

线程池的execute()和submit()方法有什么区别

  • execute是Executor接口的方法，submit是ExecuteService接口的方法
  • execute的参数是Runnable，用于执行不需要返回值的任务，无法判断线程是否执行成功。submit的参数可以是Runnable、Callable，用于执行需要返回值的任务，返回一个Future对象，通过get()获取返回值，可以判断任务是否执行成功
  • 方法中抛出异常，execute会直接抛出异常，submit会在get()获取结果的时候抛出异常，如果不获取结果，submit不抛出异常

线程池的ctl变量

​ private final AtomicInteger ctl = new AtomicInteger(ctlOf(RUNNING, 0)); //默认状态为RUNNING，线程数量为0
​ ctl是一个原子类变量，包含两部分：线程池状态runState (五种)，线程数量workerCount，ctl通过分位来表示状态和数量，其中高3位为runState，低29位为workerCount。当线程池增加和销毁线程时，会通过CAS更新ctl。

线程的状态

  • 操作系统的五种状态
    创建、就绪、运行、阻塞、终止
  • 应用程序中的六种状态
    • New(新建) 线程被创建，但未启动，new MyThread()
    • Runnable(就绪态) 调用t.start()方法，等待CPU的调度
    • Blocked(阻塞态) 线程请求对象锁但被其他线程持有进入阻塞态，当线程获得锁时，变为就绪态
    • Waiting(无限等待) 线程调用wait方法进入无限等待，需要其他线程notify才能唤醒
    • Timed Waiting(计时等待) Thread.sleep()
    • Teminated(终止态) run方法执行完成或异常退出

线程池核心线程数怎么设置呢

CPU密集型：N+1
IO密集型：2*N

悲观锁

​ 共享资源每次只给一个线程使用，其它线程阻塞，用完后再把资源转让给其它线程，synchronized和ReentrantLock是悲观锁的体现，用于多写场景。

乐观锁

​ 认为共享资源每次被多个线程访问的时候不会出现问题，无需加锁，只在提交修改的时候去验证对应的资源是否被其它线程修改，如CAS，用于多读场景。

CAS

​ 基于硬件级别的指令实现的同步原语，JUC中许多同步类基于CAS构建

​ CAS包含三个操作数-内存位置值(V)、旧的预期值(A)、新值(B)。首先取出内存位置值，如果内存位置值(V)与旧的预期值(A)相同，那么处理器会自动将内存位置值替换成新值(B)。如果CAS执行失败，则会重新取内存中的值，继续CAS，直到成功，如果竞争激烈，则会竞争次数过多。

​ CAS存在的问题ABA问题：CAS过程中其他线程将变量从A修改为B,再将B修改为A,CAS判断没变化执行操作，实际上值是被修改的，与设计原语不符。Automic包中引入AtomicStampedReference类来解决这个问题。每次更新时将变量的版本号+1，之前的ABA问题，版本号就会变成3，从而解决了ABA。

中断机制

JMM (保证线程安全)

​ JMM概念：Java内存模型规定所有变量都存储在主内存，主内存是共享的内存区域，所有线程都可以访问。线程对变量的读写操作必须在本地内存中进行，本地内存存储了主内存的副本，不同线程之间无法访问对方的本地内存，线程间的通信必须通过主内存完成。

​ JMM同步的规定

  • 线程加锁前，需要把主内存中的最新值加载到线程私有的本地内存中
  • 线程解锁前，需要把线程私有的本地内存中的最新值更新到主内存中

​ 三大特性：

  • 可见性：当一个线程对主存中的共享变量做了修改，其他线程可以立即看到最新值
  • 原子性：可以借助原子类利用CAS，也可以借助synchronized和lock对程序加锁
  • 有序性：使用volatile禁止指令重排序，保证代码的执行顺序

ThreadLocal

Thread类的部分方法和成员变量

每个线程内部都维护着一个threadLocals变量，该变量中存储着Enty[]类型的table变量，table是一种数组类型的HashMap，存储[ThreadLocal<T>, value]键值对

调用get()，获取当前线程对应的threadLocals，根据key：users查找value:user

ThreadLocal对象是弱引用，在gc的时候，该对象会被回收。ThreadLocal对象只是作为ThreadLocalMap的一个key而存在的，现在key在堆内存中被回收了，key值变成了null，但是它对应的value并没有被回收而且无法被访问，导致内存泄露。针对这一问题，ThreadLocalMap类在每次调用set()的时候，会调用cleanSomeSlots()自动清理key为null的value，如此value也能被回收了。

InheritableThreadLocal实现跨线程数据同步

  • 当线程创建新的线程时new Thread()，会调用Thread类中的init()方法将当前线程中的inheritableThreadLocal变量复制到新创建线程的inheritableThreadLocal中，从而实现跨线程数据共享

AQS

AQS 全程为 AbstractQueuedSynchronizer ，是并发编程中的核心组件，作为ReentrantLock，CountDownLatch的底层实现

AQS有两种锁机制

  • 共享锁：同一时刻允许多个线程同时获得锁，在CountDownLatch中实现
  • 独占锁：同一时刻只允许一个线程获得锁，在ReentrantLock中实现

AQS设计

  • 维护了volatile修饰的state字段，代表锁的使用情况以及锁的重入次数
  • 使用CAS对state字段进行原子性修改
  • 维护了线程等待队列，由内部类Node形成的一个双向链表，Node中存储了线程的信息、状态、前后节点

ReentrantLock流程

  • 线程首先调用rentrant.lock()，调用tryAcquire()中的cas(state)获取锁资源
  • 如果其他线程在锁未释放时便执行lock()，会将该线程加入到等待队列 Node双链的队尾
  • 调用acquireQueued()中的park()进入阻塞状态
  • 获取锁资源的线程调用unlock()后，调用tryRelease()使state-1，当state为0时如果等待队列不为空则唤醒 unpark() 队列中最前面未放弃的线程并获取锁

ReentrantLock：可重入锁

  • 公平锁：线程会直接放置到AQS阻塞队列的末尾，从阻塞队列头部唤醒线程
  • 非公平锁：新来的线程会尝试进行CAS计算，成功则直接获取锁; 失败则被放到阻塞队列末尾

AQS中未实现的方法tryAcquire()，tryRelease()，tryAcquireShared()，tryReleaseShared() 它们的在ReentrantLock内的FairSync和NoFairSync中实现的

分治策略和工作窃取

分治策略：将大任务分成若干小任务，之后再并行对这些小任务进行计算，最终汇总这些任务的结果，得到最终的结果

工作窃取：每个线程维护一个WorkQueue (双端队列)，当有线程的任务队列中没有任务时，会从其他线程的任务队列的尾部去窃取任务执行

直接调用 Thread 类的 run 方法

​ 调用 start() 方法使线程进入就绪状态，直接执行 run() 方法的话，会把 run() 方法当成一个 main 线程下的普通方法去执行，并不会在某个线程中执行它，不是多线程工作。

WAITING和BLOCKED区别

​ BLOCKED是指当多个线程在竞争Synchronized同步锁时，若锁已被占用无法竞争到资源，线程会被阻塞进入BLOCKED状态。当线程可以获取资源时，自动从阻塞态中被唤醒，等待CPU的调度

​ WAITING是指在程序中调用Object.wait()方法时进入等待状态，在该状态下会释放CPU和已获得的锁，当有其他线程调用Object.notify()方法时才会被唤醒进入就绪态，等待CPU的调度

接口隐藏

Guava包限流类RateLimiter

  • 计数器法：在一段时间间隔内，处理请求的最大数量固定，超过部分不做处理，手写
  • 漏桶算法：以恒定的时间间隔从桶中获取资源，不允许某种程度的突发传输
  • 令牌桶算法：存放一个固定容量令牌的桶，安装固定的速率往桶里添加令牌，而如果请求需要被处理，则需要先从桶里获取一个令牌，当桶里没有令牌可取时，则拒绝服务。当桶满时，新添加的令牌被丢弃或拒绝。只要令牌桶中存在令牌就允许某种程度的突发传输。限流的方式分为两种，一种是如果不能拿到令牌直接返回false，一种是拿不到令牌会阻塞一段时间看能不能拿到。

后端如何防止订单重复提交 (幂等性问题)

  • 加上了秒杀接口地址隐藏之后可以防止恶意用户登陆之后，通过不断调用秒杀地址接口来骚扰服务器，所以要动态获取秒杀地址，只有真正点击秒杀按钮，才会根据用户id和商品id生成对应的秒杀接口地址
  • 将需要处理的数据存放到redis的set中防重，每次处理数据时，判断该数据在set中是否存在
  • 将订单表中的订单编号字段设置为唯一索引，插入成功则不重复
