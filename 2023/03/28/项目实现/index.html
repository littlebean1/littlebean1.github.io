项目实现

刷新issue

  • 首先调用Codehub的批量下载接口，下载每个项目对应的issue数据到excel中
    • 由于调用批量下载接口时其内部需要调用相应的接口把数据下载到excel中，因此在业务中调用接口获取excel时需要等待数据成功下载到excel中在读取excel文件，此处需要Thread.sleep(Constants.DELAY_TIME)睡一会等待数据下载完成

      • 由于该方式要阻塞主线程，因此我们选择创建一个线程Thread-1，让该线程负责阻塞并读取excel中的数据。

      • 然后我们再创建另外的线程Thread-2将mysql中存储的对应项目issue数据读取出来

      • 等到Thread-1与Thread-2都完成后，遍历excel中读取的数据，判断哪些需要更新，哪些需要添加，使用三个线程，分别进行批量插入与批量修改数据库，使用hset修改redis的hash类型的数据 (若field不存在则插入，若存在的更新)，三个线程执行完成后继续遍历其他项目。

      • 使用CompletableFuture异步执行任务，优化代码性能。

        • CompletableFuture使用了默认的ForkJoinPool线程池来执行任务，ForkJoinPool线程池中的核心线程数为内核数-1。也可以使用Executors.newFixedThreadPool()方法来创建一个固定数量的线程池，以便更精确地控制线程的数量

        • ExecutorService executor = Executors.newFixedThreadPool(5);
CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> {
    // 任务逻辑
}, executor);

单点登录

  • 单点登录分为同域单点登录和跨域单点登录

  • 访问服务1：

    • 用户访问服务1，服务1判断根据access_token判断redis存储的用户登陆状态，如果没有存储则检测到请求参数中有没有ST，若没有则将请求内部重定向到SSO系统，SSO系统检查cookie中有没有key为TGC的数据 (SSO根据TGC可获取TGT)，SSO系统判断没有TGC则跳转进SSO系统的登录页，参数带上访问服务1的目标地址

    • 用户访问登录页填写登录表单，SSO判断表单数据是否在系统中，存在则是生成 TGT(包含用户信息) 存储到redis中和SSO系统的cookie中，随机生成ST存储到redis中并作为参数发送给客户端进行重定向 ( cookie(TGC : tgt-123) tgt-123 : {username,userid,expireTime…} ST-1234 : {tgt-123, username,expireTime…} )

    • 用户拿到ST访问服务1，服务1判断redis中存储的用户登陆状态，若没有则将请求内部转发到SSO系统(实际上是用httpclient发出一个http请求)，SSO系统判断ST是否合法(存在于redis中)，若合法则取出ST的value中存储的用户的相关信息(称为access_token)，将ST从redis中删除，刷新tgt-123的过期时间，将生成的access_token存储到redis的set集合中 (tgt-123-access_token: {access_token})，此时服务1将用户登录状态(access_token)写入redis中(string: {access_token_key:access_token }) ，下次访问时直接判断redis中的access_token即可

  • 访问服务2：

    • 用户访问服务2，服务2根据access_token判断redis存储的用户登陆状态，如果没有存储则检测到请求参数中有没有ST，若有则将请求内部重定向到SSO系统，SSO系统检查cookie中有没有key为TGC的数据 (SSO根据TGC可获取TGT)，SSO系统判断该用户已登陆，随机生成ST存入redis，将ST返回给用户并带上访问服务2的目标地址
    • 用户拿到ST访问服务2，服务2判断判断redis中存储的用户登陆状态，若没有则将请求内部转发到SSO系统(实际上是用httpclient发出一个http请求)，SSO系统判断ST是否合法，若合法则返回成功，此时服务2将用户登录状态(access_token)写入redis中，下次访问时直接判断redis中的access_token即可

  • refreshToken机制：在SSO生产access_token的同时，会生成一个refresh_access_token存储到redis中，时长为access_token的两倍。当用户在访问服务A判断服务A的access_token过期时，会取出refresh_access_token去SSO系统中生产新的access_token存储到redis的tgt-123集合中，将tgt-123过期时间刷新，并将access_token中的用户信息返回到服务A中

  • 参数声明

    • TGT （Ticket-Granting Ticket）：由SSO系统生成，表明当前用户存储在SSO服务器上，TGT中存储了用户信息，过期时间2h，过期后需要重新登录认证
    • ST （Service Ticket）：随机生成，ST用来验证用户身份，用户在SSO系统获得ST后，将ST发送给需访问的服务，SSO服务验证ST的有效性，用过一次即过期，默认过期时间10s，如果过期可以重新访问SSO让SSO重新生成ST
    • 顶级域名：.huawei.com，下级域名：a.huawei.com，b.huawei.com
    • 单机情况下通常只用session即可存储用户登录信息，分布式情况下需要借助redis存储用户登录信息

  • 单点登录注销

  • 用户发出注销请求到服务A，由于用户已登录进入注销接口，在该接口中服务A将注销请求转发到SSO系统中，SSO系统删除自身cookie中的TGC:tgt-123, 删除redis中存储的tgt-123，通知用户已经登录的系统，即tgt-123-access_token中存储的token，拿到token中存储的url信息 (之前登录时存储的重定向url)，对这些url在服务器内部发出httpclient请求，当所有的客户端清除掉自身存储的access_token后退出登录成功，此时SSO系统将请求转发到服务A中，服务A判断未登录将请求转发到SSO系统的登录页
